# Правила переходов между этапами

Документ фиксирует фактические условия смены этапа в текущей реализации.

## Список этапов

- `PREPARING`
- `LABOR`
- `AFTER_BIRTH`

## Главный принцип

Система не переключает этап автоматически ни по ПДР, ни по статистике схваток, ни по истории событий.

Автоматических переходов по времени нет.

## Источники смены этапа

### 1. Ручное переключение из drawer

Источник:

- `AppNavHost`

Условие:

- пользователь выбирает этап в чипах drawer.

Результат:

- `Settings.appStage` меняется на выбранный;
- все зависимые экраны перестраиваются.

### 2. Ручное переключение из настроек

Источник:

- `SettingsScreen`
- `SettingsViewModel`

Условие:

- пользователь выбирает этап на экране `Настройки` и сохраняет настройки.

Результат:

- система переходит в выбранный этап.

### 3. Переход в `LABOR` через действие “Начались роды”

Источник:

- `DashboardScreen`
- `EventsScreen`
- `MarkLaborStartedUseCase`

Условие:

- пользователь нажимает кнопку `Начались роды`.

Действие:

1. `Settings.appStage` меняется на `LABOR`.
2. Если `laborStartTime` еще не был задан, он записывается текущим временем.
3. Если `laborStartTime` раньше отсутствовал, в журнал добавляется событие `LABOR`.

Результат:

- приложение входит в сценарий родов.

### 4. Переход в `AFTER_BIRTH` через действие “Ребенок родился”

Источник:

- `EventsScreen`
- `MarkBirthUseCase`

Условие:

- пользователь подтверждает событие рождения.

Действие:

1. `Settings.appStage` меняется на `AFTER_BIRTH`.
2. Если `birthTime` раньше не был задан, записывается текущее время.
3. Если переданы необязательные данные, они сохраняются.
4. Если это первое сохранение рождения, в журнал добавляется событие `BIRTH`.

Результат:

- приложение входит в послеродовой сценарий.

### 5. Переход из экрана “В родах” после сохранения summary

Источник:

- `LaborScreen`
- `LaborViewModel.saveSummary()`

Условие:

- пользователь нажимает `Сохранить данные родов`.

Правило:

- если после сохранения `birthTime != null`, этап становится `AFTER_BIRTH`;
- иначе если `laborStartTime != null`, этап становится `LABOR`;
- иначе этап не меняется.

Результат:

- агрегированные данные о родах синхронизируют этап с текущим смысловым состоянием.

## Чего система не делает

- не переводит в `LABOR`, если ПДР близко;
- не переводит в `LABOR`, если схватки стали регулярными;
- не переводит в `AFTER_BIRTH`, если пользователь просто начал использовать трекеры;
- не возвращает этап назад автоматически;
- не завершает `AFTER_BIRTH` по таймеру.

## Значение действия “Начались роды”

В текущей реализации это не медицинское определение и не автоматический диагноз.

Это ручной продуктовый триггер для перевода интерфейса в более интенсивный сценарий. Пользователь должен нажимать его в тот момент, когда:

- подготовительный режим больше не актуален;
- нужно поднять наверх события, схватки, выезд и SOS;
- пользователь осознанно переходит в сценарий родов.

## Значение этапа “После родов”

Этап `AFTER_BIRTH` сейчас не ограничен по времени.

Он длится до тех пор, пока пользователь:

- не переключит этап вручную;
- или не выполнит полный сброс локальных данных.

## Таблица переходов

### `PREPARING -> LABOR`

Допустимые триггеры:

- ручной выбор этапа;
- `Начались роды`;
- `Сохранить данные родов`, если задано `laborStartTime` и не задано `birthTime`.

### `PREPARING -> AFTER_BIRTH`

Допустимые триггеры:

- ручной выбор этапа;
- `Ребенок родился`;
- `Сохранить данные родов`, если задано `birthTime`.

### `LABOR -> AFTER_BIRTH`

Допустимые триггеры:

- ручной выбор этапа;
- `Ребенок родился`;
- `Сохранить данные родов`, если задано `birthTime`.

### `LABOR -> PREPARING`

Допустимые триггеры:

- только ручной выбор этапа.

### `AFTER_BIRTH -> LABOR`

Допустимые триггеры:

- только ручной выбор этапа.

### `AFTER_BIRTH -> PREPARING`

Допустимые триггеры:

- только ручной выбор этапа;
- косвенно после полного сброса данных, когда настройки снова становятся дефолтными.
