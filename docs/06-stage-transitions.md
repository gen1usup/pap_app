# Правила переходов между этапами

Документ фиксирует фактические правила смены этапа в текущей реализации.

## Список этапов

- `PREPARING`
- `CONTRACTIONS`
- `AT_HOSPITAL`
- `AT_HOME`

## Главный принцип

Переходы не выполняются автоматически по ПДР, статистике схваток или времени.

Пользователь либо:

- переключает этап вручную;
- либо вызывает явное действие, которое меняет этап через use case.

## Источники смены этапа

### 1. Активация этапа из drawer

Источник:

- drawer в shell-навигации;
- экран [StageDetailsScreen.kt](c:/PROJECTS/pap_app/app/src/main/java/com/dadnavigator/app/presentation/feature/stages/StageDetailsScreen.kt)

Шаги:

1. Пользователь открывает drawer.
2. Выбирает этап в группе `Этапы`.
3. Открывается экран сценария этапа.
4. Пользователь нажимает `Сделать текущим этапом`.
5. `UpdateAppStageUseCase` сохраняет этап в настройках.

Результат:

- `Settings.appStage` меняется;
- `Главная`, `События` и `Чек-листы` перестраиваются.

### 2. Ручное переключение из настроек

Источник:

- `SettingsScreen`
- `SettingsViewModel`

Условие:

- пользователь выбирает этап и сохраняет настройки.

Результат:

- `Settings.appStage` меняется на выбранный этап.

### 3. Переход в `CONTRACTIONS` через `Начались роды`

Источник:

- `DashboardScreen`
- `EventsScreen`
- `MarkLaborStartedUseCase`

Действие:

1. Через `StageTransitionManager` этап становится `CONTRACTIONS`.
2. Если `laborStartTime` отсутствовал, он сохраняется.
3. Если стартовое событие еще не было добавлено, в журнал пишется событие начала родов.

Результат:

- интерфейс переходит в сценарий схваток.

### 4. Переход в `AT_HOSPITAL` через `Ребенок родился`

Источник:

- `EventsScreen`
- `MarkBirthUseCase`
- `LaborViewModel.saveSummary()` при сохранении `birthTime`

Действие:

1. В `LaborSummary` сохраняется `birthTime`.
2. При наличии сохраняются имя, вес и рост.
3. Через `StageTransitionManager` этап становится `AT_HOSPITAL`.
4. При первом сохранении рождения в журнал добавляется событие `BIRTH`.

Результат:

- приложение переходит в hospital-сценарий.

### 5. Переход в AT_HOME

Источник:

- EventsScreen
- MarkArrivedHomeUseCase
- ручной выбор этапа через StageDetailsScreen или SettingsScreen

Действие:

1. Пользователь нажимает Приехали домой на экране События в hospital-сценарии.
2. MarkArrivedHomeUseCase переводит этап через StageTransitionManager в AT_HOME.
3. В журнал добавляется запись о возвращении домой.
4. Альтернативно пользователь может вручную выбрать AT_HOME через экран этапа или настройки.

## Что система не делает

Система не делает автоматически:

- `PREPARING -> CONTRACTIONS` по близкой ПДР;
- `PREPARING -> CONTRACTIONS` по регулярности схваток;
- `AT_HOSPITAL -> AT_HOME` по таймеру;
- возврат на предыдущий этап по факту отсутствия активности.

## Совместимость со старыми данными

При чтении старых сохраненных значений используется mapping:

- `LABOR -> CONTRACTIONS`
- `AFTER_BIRTH -> AT_HOME`

Это защищает старые установки приложения от краша и сохраняет читаемость старых данных.

## Таблица допустимых переходов

### `PREPARING -> CONTRACTIONS`

Допустимые триггеры:

- ручная активация этапа;
- действие `Начались роды`;
- сохранение данных родов с заполненным `laborStartTime`, если `birthTime` еще отсутствует.

### `PREPARING -> AT_HOSPITAL`

Допустимые триггеры:

- ручная активация этапа;
- действие `Ребенок родился`;
- сохранение данных родов с заполненным `birthTime`.

### `CONTRACTIONS -> AT_HOSPITAL`

Допустимые триггеры:

- ручная активация этапа;
- действие `Ребенок родился`;
- сохранение summary с `birthTime`.

### `AT_HOSPITAL -> AT_HOME`

Допустимые триггеры:

- действие `Приехали домой` на экране `События`;
- ручная активация этапа через экран этапа;
- ручное сохранение из настроек.

### Любой этап -> `PREPARING`

Допустимые триггеры:

- ручной выбор этапа;
- полный сброс данных.
