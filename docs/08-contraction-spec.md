# Спецификация счетчика схваток

Документ описывает текущую реализацию логики и UI экрана `Счетчик схваток`.

## 1. Источники данных

Экран использует:

- активную сессию схваток;
- активную схватку внутри сессии;
- список всех схваток текущей активной сессии;
- расчетную аналитику из `CalculateContractionStatsUseCase`.

Обновление времени на экране идет раз в секунду через внутренний ticker.

## 2. Базовые сущности

### Сессия схваток

Содержит:

- `id`
- `userId`
- `startedAt`
- `endedAt`

Сессия считается активной, если:

- `endedAt == null`.

### Схватка

Содержит:

- `id`
- `sessionId`
- `userId`
- `startedAt`
- `endedAt`

Схватка считается завершенной, если:

- `endedAt != null`.

## 3. Поведение основной кнопки

### Состояние 1. Сессии нет

Условия:

- `isSessionActive == false`
- `activeContractionId == null`

Поведение:

- по нажатию создается новая сессия;
- сразу после этого создается первая схватка.

### Состояние 2. Сессия есть, но схватка сейчас не идет

Условия:

- `isSessionActive == true`
- `activeContractionId == null`

Поведение:

- по нажатию начинается новая схватка в текущей сессии.

### Состояние 3. Схватка идет

Условия:

- `activeContractionId != null`

Поведение:

- по нажатию активная схватка завершается;
- в журнал добавляется событие `CONTRACTION`.

## 4. Завершение сессии

Кнопка `Завершить сессию` отображается только при активной сессии.

Шаги:

1. Пользователь нажимает кнопку.
2. Открывается подтверждение.
3. После подтверждения сессия завершается.

Дополнительное правило:

- все незавершенные схватки в этой сессии закрываются.

## 5. Какие схватки участвуют в аналитике

Для аналитики используются только завершенные схватки:

- фильтр `endedAt != null`.

Они сортируются по `startedAt` по возрастанию.

## 6. Базовые расчетные показатели

На экране считаются:

- общее число завершенных схваток;
- средняя длительность;
- средний интервал;
- последний интервал;
- тренд;
- рекомендация;
- дополнительные внутренние показатели для принятия решения.

## 7. Окно анализа

Use case работает по recent window:

- максимум 90 минут;
- максимум 12 последних завершенных схваток.

Для анализа используется тот суффикс истории, который:

- попадает в последние 90 минут;
- и ограничен 12 последними завершенными схватками.

## 8. Условия рекомендации `PREPARE`

Для перехода в `PREPARE` должны выполняться все условия:

- в recent window минимум 4 схватки;
- минимум 3 интервала;
- средний интервал `<= 8 минут`;
- средняя длительность `>= 45 секунд`;
- паттерн держится минимум `20 минут`;
- интервалы достаточно регулярны: коэффициент вариации `<= 0.50`.

## 9. Условия рекомендации `GO_TO_HOSPITAL`

Для перехода в `GO_TO_HOSPITAL` должны выполняться все условия:

- в recent window минимум 5 схваток;
- минимум 4 интервала;
- средний интервал `<= 5 минут`;
- средняя длительность `>= 60 секунд`;
- строгий паттерн держится минимум `30 минут`;
- интервалы регулярны: коэффициент вариации `<= 0.35`;
- подряд с конца есть минимум 3 коротких интервала `<= 5 минут`;
- подряд с конца есть минимум 4 длительных схватки `>= 60 секунд`.

## 10. Когда остается `MONITOR`

Рекомендация остается `MONITOR`, если:

- недостаточно данных;
- паттерн не удерживается;
- интервалы слишком длинные;
- длительность схваток слишком короткая;
- регулярность недостаточна.

## 11. Роль тренда

Тренд считается по сравнению первой и второй половины recent window.

Варианты:

- `STABLE`
- `BECOMING_MORE_INTENSE`
- `BECOMING_WEAKER`
- `INSUFFICIENT_DATA`

Правило:

- тренд не может сам по себе привести к `GO_TO_HOSPITAL`;
- он может только помочь дойти до `PREPARE`, если базовые условия уже близки.

## 12. Поддерживаемое, но фактически не вычисляемое состояние `EMERGENCY`

В модели есть:

- `RecommendationLevel.EMERGENCY`.

Но текущий `CalculateContractionStatsUseCase` не возвращает `EMERGENCY`.

Это значит:

- UI умеет его показать;
- автоматическая аналитика сейчас его не генерирует.

## 13. Что показывается в верхней статусной карточке

Всегда отображаются:

- headline стадии;
- рекомендация текстом;
- длительность сессии;
- число завершенных схваток.

Соответствие стадий:

- `MONITOR` -> “Наблюдаем”
- `PREPARE` -> “Готовимся к выезду”
- `GO_TO_HOSPITAL` -> “Пора ехать”
- `EMERGENCY` -> “Срочно нужна помощь”

## 14. Что показывается в метриках

Первая строка:

- средний интервал;
- средняя длительность.

Вторая строка:

- последний интервал;
- тренд.

Если данных нет:

- показывается `Не указано`.

## 15. Мини-график

Источником для графика служат завершенные схватки.

Показываются:

- максимум 6 последних завершенных схваток.

Особенности:

- высота бара пропорциональна длительности;
- минимальная визуальная высота ограничена коэффициентом `0.15`, чтобы короткие схватки были видны;
- подпись под каждым баром — длительность в секундах.

## 16. История схваток

Источник:

- только завершенные схватки.

Порядок:

- от новых к старым.

Для каждой карточки показывается:

- порядковый номер;
- дата и время начала;
- длительность;
- интервал до предыдущей схватки.

Интервал вычисляется как разница между `startedAt` текущей и следующей по списку более ранней схватки.

## 17. Побочные эффекты на другие части приложения

После завершения каждой схватки:

- в журнал добавляется событие `CONTRACTION`.

На главной:

- если этап `PREPARING` и сессия уже активна, shortcut счетчика остается видимым;
- если этап `AFTER_BIRTH`, shortcut на главной скрывается независимо от истории.

## 18. Что счетчик не делает автоматически

- не меняет `AppStage`;
- не вызывает `SOS`;
- не переводит в `LABOR`;
- не завершает роды;
- не определяет медицинский диагноз.
