# Детализированные Use Cases

Документ описывает текущую реализацию сценариев в кодовой базе, а не абстрактное желаемое состояние.

## UC-01. Первый запуск после чистого сброса

Цель:

- открыть приложение в дефолтном состоянии.

Предусловия:

- локальные данные отсутствуют или были полностью очищены;
- приложение запущено после очистки.

Основной поток:

1. Пользователь открывает приложение.
2. Система читает настройки из DataStore.
3. При отсутствии настроек используются значения по умолчанию.
4. Текущий этап считается `PREPARING`.
5. Главная открывается в подготовительном сценарии.

Конечный результат:

- приложение открывается без критических ошибок;
- этап равен `PREPARING`;
- интерфейс не показывает послеродовой и боевой родовой контекст, пока он явно не активирован.

## UC-02. Сохранить ПДР и имя пользователя

Цель:

- сохранить персональные настройки, влияющие на контекст и тексты.

Предусловия:

- пользователь открыл экран `Настройки`.

Основной поток:

1. Пользователь вводит имя.
2. Пользователь вводит ПДР.
3. Нажимает `Сохранить настройки`.
4. Система валидирует дату.
5. Система сохраняет `fatherName`, `dueDate`, тему, уведомления, адрес роддома и этап.
6. Обновленные настройки становятся источником правды для UI.

Альтернативные потоки:

- если дата не проходит `LocalDate.parse`, показывается ошибка формата даты;
- если поле даты пустое, `dueDate` сохраняется как `null`.

Конечный результат:

- настройки сохранены в DataStore;
- snapshot настроек записан в Room;
- главная показывает карточку ПДР, если дата задана.

Изменения данных:

- `Settings`
- `settings`
- `users`

## UC-03. Вручную переключить этап приложения

Цель:

- изменить сценарный режим приложения без автоматических триггеров.

Предусловия:

- пользователь открыл drawer или экран `Настройки`.

Основной поток:

1. Пользователь выбирает один из этапов: `PREPARING`, `LABOR`, `AFTER_BIRTH`.
2. Система вызывает сохранение нового `Settings.appStage`.
3. Все экраны, подписанные на настройки, автоматически перестраиваются.

Конечный результат:

- интерфейс переходит в выбранный этап;
- главная, чек-листы и сценарные экраны пересчитывают свой контекст.

Изменения данных:

- `Settings.appStage`
- `settings.appStage`

## UC-04. Отметить “Начались роды”

Цель:

- перевести приложение из подготовки в родовой сценарий.

Предусловия:

- пользователь находится на главной или в разделе `События`;
- пользователь осознанно решил перейти в сценарий родов.

Основной поток:

1. Пользователь нажимает `Начались роды`.
2. Система переводит `Settings.appStage` в `LABOR`.
3. Система читает `labor_summary`.
4. Если `laborStartTime` еще не задан, записывает текущее время как начало родов.
5. Если `laborStartTime` раньше отсутствовало, добавляет в журнал событие типа `LABOR`.
6. Главная и `События` перестраиваются под родовой сценарий.

Альтернативные потоки:

- если время начала родов уже сохранено, оно не затирается повторным нажатием.

Конечный результат:

- текущий этап равен `LABOR`;
- на главной повышается приоритет `Событий`, схваток, вод и SOS;
- в журнале есть событие начала родов, если его раньше не было.

Изменения данных:

- `Settings.appStage = LABOR`
- `labor_summary.laborStartTime`
- `timeline_events` типа `LABOR`

Примечание:

- текущее приложение трактует `Начались роды` как ручной пользовательский триггер, а не как автоматический медицинский вывод.

## UC-05. Начать и завершить схватку

Цель:

- зафиксировать одну схватку внутри текущей сессии.

Предусловия:

- открыт экран `Счетчик схваток`.

Основной поток:

1. Пользователь нажимает большую круглую кнопку.
2. Если активной сессии нет, система автоматически создает новую сессию.
3. Если активной схватки нет, система создает новую схватку с `startedAt = now`.
4. Пользователь повторно нажимает кнопку в конце схватки.
5. Система записывает `endedAt` для активной схватки.
6. В журнал добавляется событие типа `CONTRACTION`.
7. Аналитика и история пересчитываются.

Альтернативные потоки:

- если активная схватка уже идет, повторное нажатие завершает именно ее;
- при ошибке записи показывается snackbar с общей ошибкой.

Конечный результат:

- схватка сохранена с началом и окончанием;
- статистика схваток обновлена;
- история и мини-график отображают новую запись.

Изменения данных:

- `contraction_sessions`
- `contractions`
- `timeline_events` типа `CONTRACTION`

## UC-06. Завершить сессию схваток

Цель:

- явно закончить период наблюдения за схватками.

Предусловия:

- существует активная сессия схваток.

Основной поток:

1. Пользователь нажимает `Завершить сессию`.
2. Открывается подтверждающий диалог.
3. Пользователь подтверждает действие.
4. Система завершает текущую сессию текущим временем.
5. Все незавершенные схватки в этой сессии также закрываются.

Конечный результат:

- активной сессии больше нет;
- круглая кнопка возвращается в стартовое состояние;
- история завершенных схваток сохраняется.

## UC-07. Зафиксировать отхождение вод

Цель:

- начать отсчет после отхождения вод и сохранить параметры события.

Предусловия:

- открыт экран `Таймер вод`.

Основной поток:

1. Пользователь выбирает цвет вод.
2. При необходимости вводит заметку.
3. Нажимает `Сохранить событие`.
4. Система создает запись `water_break_events`.
5. Система добавляет в журнал событие `WATER_BREAK`.
6. Экран начинает показывать активный отсчет.

Конечный результат:

- в системе есть активное событие отхождения вод;
- история вод обновлена;
- главная и `События` могут поднять контекст по водам.

Изменения данных:

- `water_break_events`
- `timeline_events` типа `WATER_BREAK`

## UC-08. Закрыть активное событие вод

Цель:

- завершить активный отсчет после вод.

Предусловия:

- существует активное событие вод.

Основной поток:

1. Пользователь нажимает `Завершить активное событие`.
2. Система записывает `closedAt` для активной записи.
3. Карточка статуса перестает быть активной.
4. Запись остается в истории.

Конечный результат:

- активный таймер вод завершен;
- историческая запись не удаляется.

## UC-09. Отметить выезд и прибытие в роддом

Цель:

- быстро фиксировать логистические события.

Предусловия:

- открыт экран `События`.

Основной поток:

1. Пользователь нажимает `Выехали в роддом`.
2. Система добавляет событие `LABOR` в журнал.
3. Позже пользователь нажимает `Приехали в роддом`.
4. Система добавляет второе событие `LABOR`.

Конечный результат:

- хронология логистики сохранена в журнале.

## UC-10. Отметить “Ребенок родился”

Цель:

- зафиксировать рождение и перевести приложение в послеродовой этап.

Предусловия:

- открыт экран `События`;
- пользователь находится в контексте родов или хочет вручную перейти к послеродовому сценарию.

Основной поток:

1. Пользователь нажимает `Ребенок родился`.
2. Открывается нижний sheet.
3. Пользователь может заполнить имя, вес и рост.
4. Нажимает `Сохранить событие` либо `Сохранить без деталей`.
5. Система валидирует вес и рост, если они введены.
6. Система переводит `Settings.appStage` в `AFTER_BIRTH`.
7. Система сохраняет `birthTime` и необязательные поля.
8. Если рождение фиксируется впервые, добавляет событие `BIRTH` в журнал.
9. Sheet закрывается.

Альтернативные потоки:

- если вес или рост заполнены некорректно, показывается ошибка и сохранение не выполняется;
- если `birthTime` уже был сохранен, новое время не затирает старое.

Конечный результат:

- текущий этап равен `AFTER_BIRTH`;
- время рождения сохранено;
- главная перестроена под послеродовой сценарий.

Изменения данных:

- `Settings.appStage = AFTER_BIRTH`
- `labor_summary`
- `timeline_events` типа `BIRTH`

## UC-11. Дозаполнить данные родов на экране “В родах”

Цель:

- сохранить или уточнить агрегированные данные родов.

Предусловия:

- открыт экран `В родах`.

Основной поток:

1. Пользователь при необходимости нажимает `Отметить сейчас` для начала родов.
2. При необходимости нажимает `Время рождения`, чтобы подставить текущее время рождения.
3. Заполняет имя, вес и рост.
4. Нажимает `Сохранить данные родов`.
5. Система валидирует числовые поля.
6. Система сохраняет `labor_summary`.
7. Если раньше отсутствовал `laborStartTime`, создается событие `LABOR`.
8. Если раньше отсутствовал `birthTime`, создается событие `BIRTH`.
9. Если `birthTime != null`, этап становится `AFTER_BIRTH`, иначе при наличии `laborStartTime` этап становится `LABOR`.
10. Пользователь получает snackbar с подтверждением сохранения.

Конечный результат:

- агрегированные данные родов сохранены;
- этап синхронизирован с ключевыми временными отметками;
- журнал дополнен недостающими событиями.

## UC-12. Открыть и использовать чек-листы

Цель:

- просматривать и отмечать релевантные списки по этапу.

Предусловия:

- пользователь открыл экран `Чек-листы`.

Основной поток:

1. При первом открытии для пользователя выполняется сидинг базовых чек-листов.
2. Экран определяет эффективный этап:
   выбранный вручную на экране или текущий `Settings.appStage`.
3. Отображаются только чек-листы этого этапа.
4. Пользователь отмечает нужные пункты.
5. Прогресс пересчитывается автоматически.

Альтернативные потоки:

- пользователь переключает этап через chip-фильтры;
- если для этапа нет чек-листов, показывается пустое состояние.

Конечный результат:

- прогресс по этапу сохранен;
- экран показывает только релевантные группы.

## UC-13. Добавить пользовательский пункт в чек-лист

Цель:

- дополнить системный шаблон собственной задачей.

Предусловия:

- открыт конкретный чек-лист на экране `Чек-листы`.

Основной поток:

1. Пользователь вводит текст в поле `Новый пункт`.
2. Нажимает `Добавить пункт`.
3. Система создает новый `checklist_item` для текущего checklist.
4. Поле ввода очищается.

Конечный результат:

- в текущем чек-листе появляется пользовательский пункт.

## UC-14. Просмотреть журнал и отфильтровать события

Цель:

- быстро получить хронологию нужного типа.

Предусловия:

- открыт экран `Журнал`.

Основной поток:

1. По умолчанию активен фильтр `ALL`.
2. Пользователь выбирает один из фильтров: `ALL`, `LABOR`, `POSTPARTUM`.
3. Система пересчитывает список событий.
4. Пользователь видит отфильтрованный журнал.

Правила фильтрации:

- `ALL` — все события;
- `LABOR` — `LABOR`, `BIRTH`, `CONTRACTION`, `WATER_BREAK`;
- `POSTPARTUM` — `FEEDING`, `DIAPER`, `SLEEP`, `NOTE`.

Конечный результат:

- пользователь видит ровно тот срез хронологии, который запросил.

## UC-15. Добавить ручное событие в журнал

Цель:

- сохранить произвольную запись вне жестко заданных сценариев.

Предусловия:

- открыт экран `Журнал`.

Основной поток:

1. Пользователь открывает форму добавления.
2. Выбирает тип события.
3. Вводит заголовок и описание.
4. Нажимает `Сохранить`.
5. Система сохраняет запись в `timeline_events`.
6. Пользователь получает snackbar об успехе.

Альтернативные потоки:

- если выбран тип `LABOR` и заголовок пустой, показывается ошибка `input_required`.

Конечный результат:

- новая запись появляется в журнале и участвует в фильтрах.

## UC-16. Использовать SOS и быстрые контакты

Цель:

- обеспечить максимально короткий путь к экстренным действиям.

Предусловия:

- открыт экран `SOS`.

Основной поток:

1. Пользователь видит тревожные признаки и краткие действия.
2. Может нажать `Позвонить 112`.
3. Может нажать `Позвонить в скорую / роддом`.
4. Может позвонить по одному из сохраненных быстрых контактов.
5. При необходимости открывает экран редактирования контактов.

Правила:

- быстрые контакты отображаются только если их телефон не пустой;
- кнопка `Позвонить в скорую / роддом` использует телефон `MATERNITY_HOSPITAL`, а если он пустой, используется `103`.

Конечный результат:

- пользователь быстро инициирует звонок или переходит к редактированию контактов.

## UC-17. Сохранить адрес роддома и открыть маршрут

Цель:

- иметь быстрый переход в карты без ручного поиска адреса.

Предусловия:

- открыт экран `Справка`.

Основной поток:

1. Пользователь вводит адрес роддома.
2. Нажимает `Сохранить адрес`.
3. Система сохраняет адрес в `Settings`.
4. Пользователь нажимает `Построить маршрут`.
5. Система пытается открыть `geo:` intent.
6. Если `geo:` intent недоступен, открывает web-ссылку карт.

Конечный результат:

- адрес сохранен локально;
- маршрут открывается через установленное приложение карт или браузер.

## UC-18. Использовать послеродовые трекеры

Цель:

- быстро фиксировать базовые записи после рождения ребенка.

Предусловия:

- открыт экран `Трекеры`.

Основной поток:

1. Пользователь выбирает вкладку: `Кормления`, `Подгузники`, `Сон`, `Заметки`.
2. Заполняет минимальную форму.
3. Нажимает `Добавить запись`.
4. Система валидирует ввод.
5. Сохраняет запись в профильную таблицу.
6. Дополнительно создает соответствующее событие в журнале.

Альтернативные потоки:

- если числовые поля заполнены некорректно, показывается ошибка;
- если обязательный текст для заметки пустой, запись не создается.

Конечный результат:

- запись хранится и в своей таблице, и в журнале.

## UC-19. Полный сброс локальных данных

Цель:

- вернуть приложение к почти чистому состоянию.

Предусловия:

- открыт экран `Настройки`.

Основной поток:

1. Пользователь нажимает `Сброс данных`.
2. Подтверждает действие в диалоге.
3. Система вызывает `ResetAllDataUseCase`.
4. Room очищает все таблицы.
5. DataStore очищается.
6. Экран показывает подтверждение сброса.

Конечный результат:

- локальные данные удалены;
- при следующем чтении настроек система возвращается к дефолтному сценарию.
