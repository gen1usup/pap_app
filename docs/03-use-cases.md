# Детализированные пользовательские сценарии

## UC-01. Первый запуск без ПДР

Предусловия:

- приложение запущено впервые или после полного сброса;
- ПДР не сохранена;
- этап по умолчанию `PREPARING`.

Шаги:

1. Пользователь открывает приложение.
2. Открывается `Главная`.
3. Система строит `HomeContent` на основе текущих настроек.
4. Поскольку `dueDate == null`, builder включает reminder о ПДР.
5. Пользователь видит карточку с текстом о необходимости указать ПДР.
6. Пользователь может нажать кнопку `Указать ПДР`.
7. Система открывает экран `Настройки`.

Результат:

- запуск не блокируется;
- пользователь видит ненавязчивое, но заметное напоминание;
- переход к вводу ПДР доступен в один тап.

## UC-02. Ввод ПДР

Предусловия:

- открыт экран `Настройки`;
- пользователь хочет сохранить ПДР.

Шаги:

1. Пользователь открывает блок ПДР.
2. Выбирает дату.
3. Нажимает `Сохранить настройки`.
4. `SettingsViewModel` валидирует дату.
5. При валидной дате значение записывается в `DataStore` и snapshot настроек в Room.
6. `AppViewModel` и `DashboardViewModel` получают обновленные `Settings`.
7. `Главная` перестраивается без reminder о ПДР.

Альтернативный поток:

- если дата некорректна, под полем показывается inline-ошибка, сохранение не считается успешным.

Результат:

- ПДР сохранена локально;
- главная показывает карточку срока и дней до ПДР;
- этап не меняется автоматически.

## UC-03. Переход в экран этапа из drawer

Предусловия:

- открыт один из shell-экранов: `Главная`, `События` или `Чек-листы`.

Шаги:

1. Пользователь нажимает кнопку меню.
2. Открывается drawer.
3. В группе `Этапы` пользователь выбирает нужный этап.
4. Drawer закрывается.
5. Открывается отдельный экран сценария этапа.
6. Пользователь читает описание и правила отображения.
7. Пользователь нажимает `Сделать текущим этапом`.
8. `UpdateAppStageUseCase` сохраняет новый этап.
9. Shell возвращается на `Главную`.
10. Главная перестраивается под новый этап.

Результат:

- этап меняется осознанно, а не одним случайным тапом по chip;
- пользователь видит, что именно изменит этот этап;
- навигация остается консистентной.

## UC-04. Начать роды из главной или событий

Предусловия:

- этап `PREPARING`;
- пользователь решил перейти в сценарий родов.

Шаги:

1. Пользователь нажимает `Начались роды` на `Главной` или `Событиях`.
2. Вызывается `MarkLaborStartedUseCase`.
3. Через `StageTransitionManager` этап становится `CONTRACTIONS`.
4. Если `laborStartTime` еще не было, оно сохраняется.
5. Если событие еще не создавалось, в журнал добавляется событие начала родов.
6. Зависимые экраны получают новые `Settings`.
7. Главная поднимает счетчик схваток и таймер вод выше.

Результат:

- приложение входит в интенсивный сценарий;
- история не теряется;
- интерфейс перестраивается сразу после действия пользователя.

## UC-05. Фиксация рождения ребенка

Предусловия:

- пользователь находится в сценарии родов или в экране `В родах`;
- рождение нужно зафиксировать.

Шаги:

1. Пользователь открывает экран `События`.
2. Нажимает `Ребенок родился`.
3. Открывается bottom sheet с необязательными полями имени, веса и роста.
4. Пользователь либо заполняет поля, либо пропускает их.
5. Нажимает `Сохранить событие` или `Сохранить без деталей`.
6. Вызывается `MarkBirthUseCase`.
7. В `LaborSummary` сохраняются время рождения и необязательные данные.
8. Через `StageTransitionManager` этап становится `AT_HOSPITAL`.
9. В журнал добавляется событие рождения.
10. Главная перестраивается: счетчик схваток скрывается, появляется hospital-context.

Результат:

- рождение сохранено локально;
- интерфейс переходит в сценарий `AT_HOSPITAL`;
- при неполных данных остается карточка с предложением дозаполнить сведения о ребенке.

## UC-06. Открытие журнала через top bar

Предусловия:

- пользователь находится на `Главной`, `Событиях` или `Чек-листах`.

Шаги:

1. Пользователь нажимает bell icon справа в top bar.
2. Система открывает маршрут `journal`.
3. Отрисовывается экран `Журнал`.
4. Пользователь может просматривать историю или добавить запись.

Результат:

- журнал доступен без отдельного bottom-tab;
- top-level shell остается компактным;
- навигация к истории стала вторичной, а не доминирующей.

## UC-07. Переход Главная -> Чек-листы -> Главная

Предусловия:

- приложение открыто на `Главной`.

Шаги:

1. Пользователь нажимает `Чек-листы` в нижней навигации.
2. Открывается экран чек-листов с фильтром текущего этапа.
3. Пользователь нажимает `Главная` в нижней навигации.
4. Shell выполняет top-level навигацию через единый маршрут и сохранение состояния.

Результат:

- переход работает предсказуемо;
- bottom bar не ломает back stack;
- пользователь возвращается на `Главную` без зависаний и пустого экрана.

## UC-08. Домашний сценарий после выписки

Предусловия:

- пользователь находится в AT_HOSPITAL или готов вручную завершить hospital-сценарий;
- семья возвращается домой или хочет заранее переключить интерфейс вручную.

Основной поток:

1. Пользователь открывает экран События.
2. В hospital-секции нажимает Приехали домой.
3. Вызывается MarkArrivedHomeUseCase.
4. Через StageTransitionManager этап становится AT_HOME.
5. В журнал добавляется запись о возвращении домой.
6. Главная, События и Чек-листы перестраиваются под домашний сценарий.

Альтернативный поток:

1. Пользователь открывает drawer или настройки.
2. Выбирает этап Дома.
3. Подтверждает активацию вручную.

Результат:

- на главной пропадают родовые боевые инструменты;
- поднимаются трекеры, помощь маме и домашние checklist-сценарии;
- в истории остается момент перехода домой, если действие было выполнено из Событий.
