# Use Cases

## UC-01. Первый запуск без ПДР
Предусловия:
- локальные данные пусты
- этап приложения `PREPARING`

Шаги:
1. Пользователь открывает приложение.
2. Открывается `Главная`.
3. Система строит контент этапа `PREPARING`.
4. Так как ПДР не задана, показывается ненавязчивое напоминание.
5. Пользователь может перейти в `Настройки` и задать дату.

Результат:
- приложение остается в `PREPARING`
- reminder виден до тех пор, пока ПДР не сохранена

## UC-02. Переход в `LABOR` через первую схватку
Предусловия:
- этап `PREPARING`
- активной схватки нет

Шаги:
1. Пользователь открывает экран `Активные роды`.
2. Нажимает `Начать схватку`.
3. Создается или переиспользуется активная сессия схваток.
4. Система сохраняет запись схватки.
5. Этап автоматически меняется на `LABOR`.

Результат:
- активна схватка
- этап `LABOR`
- дальнейшая рекомендация строится по текущим данным

## UC-03. Переход в `LABOR` через запись вод
Предусловия:
- этап `PREPARING`

Шаги:
1. Пользователь фиксирует воды из `Активных родов` или связанного сценария.
2. Система сохраняет запись о водах.
3. Этап автоматически меняется на `LABOR`.
4. Рекомендация пересчитывается с учетом цвета вод.

Результат:
- этап `LABOR`
- водный сценарий активен
- рекомендация обновлена

## UC-04. Фиксация рождения
Предусловия:
- текущий этап `LABOR` или пользователь вручную вернулся в него

Шаги:
1. Пользователь нажимает `Ребенок родился`.
2. Система сохраняет событие рождения.
3. Если активна схватка, она завершается.
4. Если активен таймер вод, он закрывается.
5. Этап меняется на `BABY_BORN`.
6. История схваток и сессия сохраняются.

Результат:
- этап `BABY_BORN`
- история родового сценария не потеряна
- на `Главной` сверху появляется карточка ребенка

## UC-05. Возврат в `LABOR` после рождения
Предусловия:
- рождение уже сохранено
- текущий этап `BABY_BORN`

Шаги:
1. Пользователь открывает боковое меню.
2. Переходит на preview screen этапа `LABOR`.
3. Нажимает кнопку перехода к этапу.
4. Система меняет этап на `LABOR`.

Результат:
- этап снова `LABOR`
- история схваток и событий остается доступной

## UC-06. Создание заметки
Предусловия:
- пользователь находится в любом этапе

Шаги:
1. Пользователь открывает `События`.
2. Нажимает `Добавить заметку`.
3. Заполняет заголовок и описание.
4. Нажимает `Сохранить`.
5. Система создает запись в журнале.
6. В запись автоматически пишется `stageAtCreation` текущего этапа.
7. В запись пишется `entryType = USER_NOTE`.

Результат:
- заметка видна в журнале
- заметка доступна через фильтр `Мои заметки`
- при последующей смене этапа `stageAtCreation` не меняется

## UC-07. Удаление схватки
Предусловия:
- в истории схваток есть хотя бы одна завершенная запись

Шаги:
1. Пользователь открывает `Активные роды`.
2. В списке истории выбирает корзину у нужной схватки.
3. Подтверждает удаление.
4. Система удаляет запись.
5. Производные метрики пересчитываются.

Результат:
- удаленная схватка больше не участвует в расчете интервалов и рекомендации

## UC-08. Удаление записи о водах
Предусловия:
- история вод содержит запись

Шаги:
1. Пользователь открывает экран вод.
2. Нажимает удалить у нужной записи.
3. Подтверждает удаление.
4. Система удаляет событие.
5. Рекомендация пересчитывается без удаленной записи.

Результат:
- запись исчезает из истории
- emergency или water-based recommendation может смениться на contraction-based

## UC-09. Удаление системного чек-листа
Предусловия:
- у пользователя есть системный чек-лист этапа

Шаги:
1. Пользователь открывает `Чек-листы`.
2. Удаляет нужный чек-лист.
3. Система помечает его удаленным.
4. Позже экран чек-листов открывается снова или приложение перезапускается.

Результат:
- чек-лист не показывается
- seed-логика не создает его заново автоматически
